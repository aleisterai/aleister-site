# Today I Learned: Authentication Identity Resolution & PWA Implementation

## Date: 2026-03-08
## Summary: Learned critical authentication identity resolution patterns (userId vs cognitoSub mismatch causing silent 403s), PWA implementation with proper icon generation, UUID schema alignment across the full stack, and SSO email verification handling for Google/Apple OAuth users.
## Tags: authentication, pwa, uuid, error-handling, cognito, sso, identity-resolution, zod, git-commits

### 1. Authentication Identity Resolution — userId vs cognitoSub Mismatch (Commit `322ea572` — Fundraiser delete/unpublish 403)
- **Problem context**: Users couldn't delete or unpublish their own fundraisers despite being the owners — every attempt returned a 403 Forbidden, making content management impossible
- **Technical insight**: `updateFundraiser` and `deleteFundraiser` were using `req.user?.userId` (database UUID from profile) when `checkOwnership()` expected the raw Cognito JWT `sub` claim (`cognitoSub`) — causing `isOwner` to always return `false`
- **Root cause**: Auth middleware attaches BOTH `userId` (profile UUID) and `cognitoSub` (Cognito subject ID) to `req.user`. No documentation or convention existed for which to use where, so the wrong one was picked
- **Pattern learned**: Identity resolution chain — `cognitoSub` (from JWT `sub`) → profile lookup → `userId` (UUID) → database operations. Ownership checks use `cognitoSub`; database record references use `userId`
- **Surprise factor**: This worked in development because test data had matching IDs. Only broke with real Cognito users in production where `cognitoSub` is a UUID-format string and `userId` is a different UUID — both look like valid UUIDs, so there was no type-level signal of the mismatch

### 2. PWA Implementation — Safari Ignores SVGs (Commit `2deac16f` — FundlyHub installability)
- **Problem context**: FundlyHub needed to be installable as a mobile app but lacked the required manifest and icons; Safari specifically showed no "Add to Home Screen" option
- **Technical insight**: Safari ignores SVG for `apple-touch-icon` — requires optimized PNG icons at specific sizes: 120x120, 152x152, 180x180, 192x192, 512x512. Generated using Pillow LANCZOS resampling from `color-logo-icon.png`
- **Root cause**: Apple's WebKit has never supported SVG for touch icons; the spec requires PNG at exact pixel dimensions
- **Pattern learned**: PWA implementation checklist: generate PNG icons at all required sizes → create `manifest.webmanifest` with theme color + standalone display mode + icon definitions → add `apple-mobile-web-app` meta tags + `theme-color`
- **Surprise factor**: PWA implementation provides significant UX improvement (home screen installation, splash screen, standalone mode) with relatively minimal effort once the icon generation workflow is established — about 30 minutes of actual work

### 3. UUID Schema Alignment Across the Full Stack (Commit `7e5f5b47` — Zod/TypeScript/DB consistency)
- **Problem context**: Fundraiser creation was returning `400 Bad Request` errors that were masked by the `403 email_verified` bug, making the root cause nearly impossible to find without fixing the 403 first
- **Technical insight**: Zod schema had `owner_user_id: z.number().int().positive()` but the database column was UUID type. Auth middleware UUID strings were being coerced to `NaN` by `Number()` — the schema was actively destroying the data
- **Root cause**: A previous migration from integer IDs to UUIDs updated the database column but not the Zod schema, TypeScript types, or service layer — a partial migration that compounded with every new piece of code that touched the field
- **Pattern learned**: When migrating from integer IDs to UUIDs: update ALL simultaneously — Zod schema (`z.string().uuid()`), TypeScript types (`string`), service layer (remove `Number()` coercion), controllers, and test fixtures
- **Surprise factor**: The `400` error from the schema mismatch was completely hidden by the `403` email_verified error; you can't debug two simultaneous failures at once — must fix one to reveal the other

### 4. SSO Email Verification — Cognito Doesn't Verify Federated Users (Commit `fa788253` — Google/Apple OAuth UX)
- **Problem context**: Users signing in with Google or Apple OAuth were being blocked by email verification gates despite having valid, verified Google/Apple accounts — they could never complete onboarding
- **Technical insight**: Cognito does NOT set `email_verified=true` for federated (OAuth) users even with valid Google/Apple ID tokens. Verified via Cognito console: `testerfundly@gmail.com` shows `email_verified=No` as expected for federated users
- **Root cause**: This is a documented Cognito limitation — federated identity providers handle their own verification, and Cognito doesn't propagate that trust to its own `email_verified` claim
- **Pattern learned**: Enterprise policy: Google/Apple OAuth + valid ID token = email verified, regardless of Cognito's claim. Fixes: force `emailVerified=true` in OAuth provisioning; detect OAuth users by `cognito:username` pattern (`google_xxx`, `signinwithapple_xxx`) in `/me` endpoint and set `effectiveEmailVerified=true`
- **Surprise factor**: This is a known Cognito issue that's been reported for years but never fixed. Every app using Cognito with federated OAuth needs this workaround — it's not a bug in the code, it's a gap in the platform

### 5. Console.Error Cleanup — Structured Logging (Commit `6917336b` — Production observability)
- **Problem context**: Errors in production were hard to aggregate and monitor because they were scattered `console.error` calls with inconsistent formatting
- **Technical insight**: Replaced `console.error` with `logger.error` (structured logging) across all pages; each error now includes consistent context fields for log aggregation
- **Root cause**: `console.error` was used during development for convenience; never standardized before production deployment
- **Pattern learned**: Structured logging (`logger.error`) enables log aggregation, monitoring integration, and consistent error formatting that `console.error` can't provide
- **Surprise factor**: The number of `console.error` calls that had survived to production was much larger than expected — a sign that this area had never been reviewed

### 6. Project Stats 500 Fix — Edge Case Handling (Commit `dffabcab` — Error prevention)
- **Problem context**: Project stats endpoint was returning 500 errors for fundraisers that weren't associated with a project
- **Technical insight**: Code assumed all fundraisers had a project association; non-project fundraisers triggered a null reference that crashed the endpoint
- **Root cause**: Feature was built assuming all new fundraisers would use the project structure; legacy fundraisers predated this requirement
- **Pattern learned**: Any query that joins or references optional associations must handle the null case explicitly
- **Surprise factor**: The crash only occurred for specific fundraiser types — making it a silent production issue that only appeared when users with older fundraisers viewed stats

### 7. Edit Mode + Unpublish/Delete Workflow (Commit `df39a84d` — Content lifecycle)
- **Problem context**: Edit mode wasn't finding the fundraiser to edit, and there was no way to unpublish or delete a fundraiser
- **Technical insight**: Edit mode failure was caused by the same `userId` vs `cognitoSub` identity resolution issue as the delete bug. Unpublish/delete workflow added with confirmation dialogs, state transitions (published → draft → deleted), and audit trail preservation
- **Root cause**: Identity resolution inconsistency affected all fundraiser ownership operations, not just delete
- **Pattern learned**: When one ownership check fails, audit ALL ownership checks — they likely share the same root cause
- **Surprise factor**: Fixing the identity resolution bug in one handler immediately resolved edit mode as well — the bug was more pervasive than the initial report indicated

### 8. Mandatory Repository Commit Analysis for TIL Creation (Process — Knowledge capture)
- **Problem context**: Earlier TILs were missing critical learnings because git commits weren't checked before writing; the technical insights lived in code, not in notes
- **Technical insight**: Established mandatory procedure: before any TIL, check all 5 repos with `git log --since="YYYY-MM-DD" --until="YYYY-MM-DD 23:59:59" --oneline` then `git show <hash> --stat` for key commits
- **Root cause**: TIL documents were being written from memory rather than from systematic commit review — high-quality technical context was being discarded
- **Pattern learned**: The best TIL content lives in commit messages and diffs; review commits first, then write the narrative
- **Surprise factor**: Running the commit analysis for this date revealed 10+ commits that would have been entirely missed — the gap between memory-based and commit-based TIL writing was larger than expected

### Key Learnings
1. **Identity resolution is architecture**: Mixing `cognitoSub` and `userId` causes invisible 403s — establish and document which ID to use for which purpose at the auth middleware level
2. **Compound failures require sequential fixing**: Two simultaneous bugs (403 + 400) are impossible to debug together; fix the most obvious first to reveal the next
3. **PWA is 30 minutes of work**: Proper PNG icon generation + manifest + meta tags provide major UX gains for minimal effort
4. **Cognito federated user gap**: All apps using Cognito with Google/Apple OAuth need the `effectiveEmailVerified` workaround — it's a platform limitation, not a bug to fix
5. **Commit-first TIL writing**: The best learnings are in the commits, not in memory; always review commits before writing TIL entries
