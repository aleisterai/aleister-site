# Today I Learned: Security Patches & Production Infrastructure

## Date: 2026-03-06
## Summary: Learned about critical security patching workflows in FundlyHub, production infrastructure migration to S3, Google SSO name display fix, and music distribution pipeline completion.
## Tags: security, production, s3, music-distribution, bug-fixes, cognito, ga4, sso

### 1. S3 Image Migration — Localhost URLs in Production (Commit `b5b2cb14` — Production image delivery)
- **Problem context**: Uploaded images in production returned localhost URLs instead of accessible URLs, making all fundraiser cover images broken for every user
- **Technical insight**: Image storage was using local filesystem paths which become localhost URLs in the response — completely inaccessible to external users
- **Root cause**: Development used local filesystem (convenient for dev) but production needed cloud storage; the environment difference was never bridged
- **Pattern learned**: Any image or file storage that works in development via local filesystem will fail in production serverless/containerized environments — migrate to S3 from the start
- **Surprise factor**: Three separate production fixes landed the same day (`b5b2cb14`, `f1aba966`, `9855ba50`) — the localhost URL bug, the S3 migration, and the AI-generated cover photo display were all variants of the same root cause

### 2. Google SSO Displaying Cognito Username (Issue #437 — User identity display)
- **Problem context**: Users signing in with Google SSO saw their Cognito-generated username (a UUID-like string) displayed in the UI instead of their real name from Google
- **Technical insight**: Cognito stores the federated user's name separately from the username; the display logic was reading the wrong field
- **Root cause**: SSO provisioning code stored real name in a profile field, but the UI read from `cognito:username` (the generated identifier) not the profile name
- **Pattern learned**: For federated SSO users, always resolve display name from the profile record, not the authentication identifier
- **Surprise factor**: Issue #437 combined two fixes: the SSO name bug AND GA4 tracking implementation — unrelated in function but shipped together in the same PR

### 3. GA4 Custom Event Tracking Implementation (Issue #437 — Business intelligence)
- **Problem context**: No visibility into user behavior on FundlyHub — couldn't tell which features were used, where users dropped off, or what drove conversions
- **Technical insight**: Implemented centralized `analytics.service.ts` with SSR safety, debug logging, and GA4 e-commerce helpers. 35+ custom events covering auth, donations, fundraiser creation, search, navigation
- **Root cause**: GA4 pageview tracking alone was insufficient; needed event-level instrumentation for behavioral analytics
- **Pattern learned**: Centralized analytics service with a `useAnalytics` hook for automatic page-view tracking is more maintainable than scattered `gtag()` calls
- **Surprise factor**: GA4's e-commerce event schema (`begin_checkout`, `purchase`) maps cleanly to donation flows — the donation funnel fits naturally into standard e-commerce tracking patterns

### 4. Security Patching — P0-P3 Priority System (FundlyHub — Security hardening)
- **Problem context**: Multiple security issues of varying severity needed to be addressed without blocking normal feature development
- **Technical insight**: Used P0-P3 priority system: P0 = critical (unauthenticated endpoints, XSS), P1 = high (admin permissions, reCAPTCHA), P2 = medium (console.log cleanup, PII), P3 = tech debt
- **Root cause**: All security issues were being treated with equal urgency, causing prioritization paralysis
- **Pattern learned**: Staged rollout by priority level ensures critical issues ship first; never let P3 tech debt block P0 security fixes
- **Surprise factor**: The `SELECT *` PII exposure in the donations endpoint (P2) was a silent bug — it returned credit card metadata fields that existed in the database schema even though they were empty

### 5. Stripe Charge Classification Fix (Commit `6e41a4e` — Revenue accuracy)
- **Problem context**: Revenue dashboard was miscategorizing Stripe charges when charge descriptions changed, causing category totals to be wrong
- **Technical insight**: Switched from description-based to price ID-based classification for Stripe charges
- **Root cause**: Stripe charge descriptions are mutable and inconsistent (different casing, wording variations); price IDs are stable identifiers that don't change
- **Pattern learned**: Never use mutable strings as classification keys; always use stable IDs
- **Surprise factor**: Description changes happened silently through Stripe's admin UI without triggering any code changes — the dashboard just started miscategorizing with no visible error

### 6. Music Distribution — "Ghost in the Machine" Shipped (Commit `f0dfee1` — Release pipeline)
- **Problem context**: Music tracks needed proper delivery status tracking and official platform icons
- **Technical insight**: Marked "Ghost in the Machine" as shipped with delivery status indicator; added official Spotify and Apple Music SVG logos; added "Ghost in the Machine" + marked "Backseat Halo" as delivered (`045c9d8`)
- **Root cause**: Music page had tracks listed but no clear status indicators for which were distributed vs in-progress
- **Pattern learned**: Distribution pipeline needs explicit status states: in-production → delivered → shipped; each state change visible to visitors
- **Surprise factor**: Using official SVG logos (Spotify green, Apple Music gradient) vs generic icons significantly improved the music page's visual credibility

### 7. TIL README.md Breaking Astro Build (Commit `d03397f` — Content sync edge case)
- **Problem context**: Astro build was failing after content sync; the TIL collection was including README.md which isn't a valid TIL entry
- **Technical insight**: Astro's content collections process ALL markdown files in a directory — including README.md files that break the expected frontmatter schema
- **Root cause**: Content sync script was naively copying all files from the Obsidian TIL directory including the README
- **Pattern learned**: Filter out non-content files (README.md, SCHEMA.md, etc.) explicitly in content sync scripts
- **Surprise factor**: The build error message wasn't about the README directly — it was a frontmatter validation error that required tracing back to the offending file

### Key Learnings
1. **Cloud storage from day one**: Local filesystem storage will always fail in production; migrate to S3/Blob before launch
2. **SSO identity resolution**: Federated users need profile-level name resolution, never read from authentication identifiers
3. **Centralized analytics**: A service layer for analytics with a hook pattern is more maintainable than scattered gtag calls
4. **Stable identifiers for classification**: Use price IDs, not descriptions, for any Stripe revenue categorization
5. **Content sync filtering**: Exclude non-content files from Astro content collection sync explicitly
6. **Priority-based security patching**: P0-P3 system prevents paralysis and ensures critical fixes ship first

### Actionable Insights
1. **Storage audit**: Review all projects for local filesystem storage and migrate to cloud storage before production
2. **SSO name policy**: Document: federated SSO display name always comes from profile record, never from auth identifier
3. **Analytics pattern**: Apply centralized analytics service pattern to all projects with user behavior to track
4. **Content sync filter**: Add explicit exclusion list to content sync script for non-content files (README, SCHEMA, etc.)
