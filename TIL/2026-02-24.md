# Today I Learned: Multi-Agent Architecture, Model Routing & iMessage

## Date: 2026-02-24
## Summary: First full day of Aleister being live. Established the multi-agent architecture, model routing strategy, iMessage integration approach, and security configuration.
## Tags: architecture, model-routing, imessage, security, github

### 1. OpenClaw Multi-Agent Architecture (System Design — Orchestration vs Standalone)
- **Problem context**: Needed an efficient way to handle diverse tasks (coding, research, writing, etc.) without managing multiple separate Discord bots
- **Technical insight**: Orchestrated subagents via `sessions_spawn` proved more efficient than standalone registered agents
- **Root cause**: Standalone agents create management overhead and fragmentation; orchestration centralizes control
- **Pattern learned**: Create specialized agent prompts (coder, researcher, writer, etc.) and dispatch from a central coordinator
- **Surprise factor**: The subagent approach was cleaner and more maintainable than expected, with better resource utilization

### 2. Model Routing Strategy (System Design — Cost vs Performance Optimization)
- **Problem context**: Needed to balance AI model performance with cost constraints across diverse task types
- **Technical insight**: Established a fallback ladder: Gemini Flash → Kimi → Sonnet → Grok → DeepSeek → Opus → GPT
- **Root cause**: Different tasks require different model capabilities; using expensive models for simple tasks is wasteful
- **Pattern learned**: Code-heavy tasks → Opus, Communications → Haiku, Balanced tasks → Sonnet, with Gemini Flash as primary
- **Surprise factor**: Gemini Flash's 1M context window and free tier eligibility made it surprisingly capable for daily operations

### 3. iMessage Integration Reality (Platform Compatibility — macOS Tahoe Challenges)
- **Problem context**: Needed reliable iMessage integration but faced compatibility issues with BlueBubbles on macOS Tahoe
- **Technical insight**: BlueBubbles requires SIP disabled for Private API access, and even then has Tahoe compatibility issues
- **Root cause**: macOS security restrictions and version incompatibilities blocked modern iMessage integration approaches
- **Pattern learned**: Legacy `imsg` CLI works reliably without SIP modifications, supporting send/read messages and emoji reactions
- **Surprise factor**: The "better" solution (BlueBubbles) was unusable, while the simpler legacy tool worked perfectly

### 4. Security Configuration (System Security — Multi-Layer Protection)
- **Problem context**: Needed to secure the system against unauthorized access while maintaining usability for authorized users
- **Technical insight**: Implemented `dmPolicy: allowlist` + `allowFrom: [userID]` to block unauthorized DMs
- **Root cause**: Open systems are vulnerable to prompt injection and unauthorized command execution
- **Pattern learned**: Multiple security gates: conversation open, execution locked with `tools.elevated.allowFrom` and `exec.security: full`
- **Surprise factor**: The layered approach provided strong security without creating friction for legitimate use

### 5. GitHub Projects Integration (Project Management — Cross-Platform Access)
- **Problem context**: Needed to integrate with existing project management workflows in FundlyHub
- **Technical insight**: FundlyHub has 222 items in production with active Supabase→AWS and MUI migrations
- **Root cause**: Project context was scattered across different systems, requiring unified access
- **Pattern learned**: Configure aleisterrai@gmail.com with project board read access for seamless integration
- **Surprise factor**: The scale of active work (222 items) highlighted the need for automated project tracking

### 6. Automation & Monitoring (System Operations — Persistence & Permissions)
- **Problem context**: Needed persistent automation (iMessage monitoring) with proper macOS permissions
- **Technical insight**: LaunchAgent enables persistent scripts, but `imsg` requires Accessibility permissions
- **Root cause**: macOS TCC (Transparency, Consent, and Control) resets permissions, requiring re-triggering
- **Pattern learned**: Cron jobs for periodic tasks combined with LaunchAgent for persistent monitoring
- **Surprise factor**: macOS permission management was more fragile than expected, requiring careful handling

### Key Learnings
1. **Architectural efficiency**: Orchestrated subagents provide cleaner management than standalone agents
2. **Cost-performance balance**: Strategic model routing optimizes both performance and expenses
3. **Pragmatic integration**: Sometimes legacy tools work better than modern "solutions"
4. **Security layering**: Multiple gates prevent attacks while maintaining usability
5. **Permission management**: macOS security requires careful, persistent configuration
