# Today I Learned: 3D Web Development & Operational Dashboards

## Date: 2026-03-05
## Summary: Learned about Three.js integration challenges, CSP configuration for 3D content, operational dashboard design, and FundlyHub testing patterns across multiple page-level unit tests.
## Tags: threejs, webgl, csp, dashboard, testing, auth, fundlyhub, mui, stripe

### 1. Three.js CSP Configuration — The Blob URL Problem (Commit `a0061a4` — 3D content rendering)
- **Problem context**: Three.js CDN integration wasn't working; scripts were being blocked and textures failed to load, leaving the 3D office scene blank
- **Technical insight**: Three.js requires `cdn.jsdelivr.net` in CSP `script-src`. Separately, Three.js uses blob: URLs internally for texture decoding — CSP was blocking these too (`17ea23e` fixed this)
- **Root cause**: CSP was correctly blocking external CDN scripts and blob: URLs as security defaults, but Three.js requires both to function
- **Pattern learned**: Blob: URL CSP restrictions block Three.js texture decoding even when the scripts themselves load correctly; both `script-src` CDN and `blob:` scheme need explicit allowance
- **Surprise factor**: The blank texture issue was a CSP problem, not a loading or format issue — adding `console.log` material debugging (`d04872f`) was the only way to distinguish CSP blocks from actual rendering failures

### 2. GLB/GLTF Texture Loading — Don't Override envMap (Commit `97cb399` — 3D model rendering)
- **Problem context**: 3D models were loading but rendering completely white — textures existed but weren't showing
- **Technical insight**: Overriding `envMap` properties on materials was wiping out the actual texture map; needed to also fix `flipY` property for correct texture orientation
- **Root cause**: Tutorial code for adding environment lighting was inadvertently overriding material texture maps, making everything white
- **Pattern learned**: When debugging 3D rendering, never override material properties wholesale — use targeted changes that preserve existing map assignments
- **Surprise factor**: The fix (`5326dbf`) was a single property removal; three commits of debugging to find one line that shouldn't exist

### 3. /office Dashboard Redesign — 24/7 Agent Monitoring (Commit `9539cdc` — Operational visibility)
- **Problem context**: The /office page was a simple static page; needed it to be a real operational dashboard showing actual agent activity
- **Technical insight**: Transformed /office into a comprehensive dashboard: agent status monitoring, task ledger with completion tracking, key metrics. Fixed agent schedule to use Pacific time instead of UTC for accurate status reporting
- **Root cause**: Original /office was decorative; operational transparency requires live data and correct timezone handling
- **Pattern learned**: Agent simulation should run 24/7 (not schedule-gated hours); always specify timezone explicitly, never assume UTC
- **Surprise factor**: The schedule-gated approach showed agents as "offline" during business hours due to UTC vs PST confusion — a timezone bug looked like a system outage

### 4. FundlyHub Page-Level Unit Tests (Issue #393 — Test coverage at scale)
- **Problem context**: FundlyHub had feature code but insufficient test coverage for page-level behavior, leaving integration issues undetected
- **Technical insight**: Added comprehensive test suite across all major pages: NotFound/ErrorRecovery (`67f92748`), SearchResults (`03c51e79`), AllCampaigns (`cb82e281`), FundraiserDetail (`50d54447`), Homepage verification (`6d03889e`)
- **Root cause**: Unit tests were added for individual components but page-level integration behavior wasn't validated
- **Pattern learned**: Page-level tests catch integration issues that component tests miss — the test names explicitly link to issue numbers for traceability
- **Surprise factor**: Writing page-level tests revealed several edge cases in error states (empty search results, malformed fundraiser IDs) that weren't visible in component-level testing

### 5. Reset Password Flow with Cognito (Commit `a9ec366b` — Auth UX)
- **Problem context**: Users had no way to recover access to accounts; no password reset flow existed
- **Technical insight**: Implemented complete Cognito verification code flow for password reset with proper UX states (request code → enter code → new password → confirmation)
- **Root cause**: Auth flow was built for happy path (login/signup) without recovery paths
- **Pattern learned**: Password reset requires multi-step state management: code request → pending → code verified → password updated — each state needs distinct UI
- **Surprise factor**: Cognito's verification code flow has strict timing constraints — codes expire, and the UX must communicate this clearly to prevent user frustration

### 6. MUI Migration for Auth Pages (Commit `514b5da4` — Design system consistency)
- **Problem context**: Auth pages used mixed styling approaches, creating visual inconsistency with the rest of the platform
- **Technical insight**: Migrated auth page components to Material-UI (MUI) for consistent design system across the platform
- **Root cause**: Auth pages were built early before MUI was adopted as the design system standard
- **Pattern learned**: Design system adoption should work backward from newest to oldest pages, not just forward into new features
- **Surprise factor**: MUI migration of auth pages uncovered several accessibility issues (contrast ratios, focus states) that the previous ad-hoc styling had missed

### 7. Stripe Checkout Production URL Fix (Commit `2a558fe` — Production environment)
- **Problem context**: Stripe checkout was redirecting users back to `localhost` after payment completion in production
- **Technical insight**: Hardcoded `localhost` in Stripe redirect URLs wasn't replaced by environment-specific values
- **Root cause**: Development convenience (hardcoded localhost) survived to production because there was no environment validation check
- **Pattern learned**: Always verify production redirect URLs for third-party services during deployment; never trust that env vars are substituted correctly
- **Surprise factor**: The checkout itself worked — users completed payment — but landed on a localhost URL that returned nothing, making it look like the entire payment failed

### Key Learnings
1. **CSP and 3D content**: Three.js needs both CDN allowances AND blob: URL permissions — half-right CSP still fails completely
2. **Debug 3D rendering with logging**: Material property logging is the only reliable way to diagnose texture issues vs CSP issues
3. **Timezone is always UTC unless specified**: Systems default to UTC; operational dashboards must specify local timezone explicitly
4. **Page-level tests complement component tests**: Integration issues between components only appear at page level
5. **Production URL validation**: Third-party redirect URLs must be validated against actual production environment before launch

### Actionable Insights
1. **CSP template**: Create a CSP template for Three.js projects covering CDN scripts and blob: URLs
2. **3D debugging checklist**: Material logging → CSP check → texture format check → flipY check
3. **Timezone standard**: Always use named timezone (America/Los_Angeles) not UTC offsets in operational code
4. **Pre-deploy checklist**: Add third-party redirect URL validation to pre-deployment checklist
