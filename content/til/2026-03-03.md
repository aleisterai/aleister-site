# Today I Learned: ClawMart Publishing & Store Development

## Date: 2026-03-03
## Summary: Learned about ClawMart API publishing workflows, security scanning importance, and e-commerce store development patterns for digital products.
## Tags: clawmart, publishing, security, e-commerce, store-development, bluebubbles, stripe, vercel-blob

### 1. ClawMart Package Publishing — Security Lesson (Platform Publishing — API key exposure)
- **Problem context**: Needed to publish the Aleister agent package to ClawMart but the CLI tool had GitHub account age restrictions (>14 days) that blocked initial uploads
- **Technical insight**: Used ClawMart API directly (`clawhub publish`) instead of CLI; API allows direct updates for paid skills. However, first upload had to be removed — `aleister-agent-package.zip` contained exposed API keys
- **Root cause**: No pre-distribution security scan step; keys were embedded in config files included in the package zip
- **Pattern learned**: Always scan packages with `grep` for API keys, secrets, and tokens before any distribution. Mandatory step, not optional
- **Surprise factor**: The package appeared clean to casual inspection — the exposed keys were in non-obvious config files, only visible with systematic scanning

### 2. ClawMart Skill Metadata Requirements (Platform Publishing — Metadata structure)
- **Problem context**: Initial ClawMart listing submission failed due to incomplete metadata fields
- **Technical insight**: ClawMart listings require specific fields: name, tagline, longDescription, category, price, productType, capabilities array, and tags array. Package versioning requires incrementing version number (e.g., 2 → 2.1.0) with correct form field name `package`
- **Root cause**: Metadata requirements weren't documented; discovered through failed submissions
- **Pattern learned**: Document platform-specific submission requirements on first attempt; don't assume standard fields
- **Surprise factor**: The `productType` field and `capabilities` array structure were non-obvious and required API response inspection to understand

### 3. BlueBubbles Server Diagnostics (System Troubleshooting — Multi-layer service debugging)
- **Problem context**: BlueBubbles appeared to be running but iMessage relay wasn't working through OpenClaw
- **Technical insight**: BlueBubbles runs multiple helper processes alongside `cloudflared` tunneling. Server can be running but OpenClaw plugin may have communication issues at a different layer
- **Root cause**: Issues can exist at: server process, cloudflared tunnel, OpenClaw plugin config, or password authentication — each layer independent
- **Pattern learned**: Use `curl -v http://localhost:1234` to verify server responsiveness; test each layer independently before assuming the issue is at the top
- **Surprise factor**: The server process was running fine; the failure was at the OpenClaw plugin authentication layer — a deeper diagnostic tool would have revealed this faster

### 4. Stripe Checkout for Digital Products (Commit `aleister-site` — Store development)
- **Problem context**: Store needed secure checkout flow for digital products with proper download delivery
- **Technical insight**: Integrated Stripe checkout with receipt pages and download flows; migrated to Vercel Blob Storage for authenticated product access
- **Root cause**: Filesystem-based delivery doesn't work in serverless deployments; Vercel functions don't persist filesystem state
- **Pattern learned**: Vercel Blob Storage provides authenticated access to paid files with proper filename handling across browsers
- **Surprise factor**: Download filename handling across browsers is genuinely complex — required RFC 5987 `filename*` parameter, URL path inclusion, AND programmatic blob downloads for iOS Safari

### 5. Dashboard Revenue Tracking Architecture (System Design — Multi-source data aggregation)
- **Problem context**: Revenue dashboard needed to show accurate real-time data from multiple sources
- **Technical insight**: Real-time revenue dashboards need multiple data sources: Stripe API, on-chain treasury data, and external price APIs (CoinGecko/CoinCap)
- **Root cause**: Single-source revenue tracking misses categories; need unified view across payment rails
- **Pattern learned**: Classify Stripe charges by price ID instead of description for reliable revenue categorization
- **Surprise factor**: Description-based classification broke silently when Stripe charge descriptions changed; price ID is stable across description variations

### 6. FundlyHub Docs Restored from Stash (Commit `dc0f584a` — Documentation recovery)
- **Problem context**: Obsidian documentation files were accidentally deleted during git operations
- **Technical insight**: Used `git stash` recovery to restore deleted files; `dc0f584a` commit "Restore deleted Obsidian docs and other files from stash"
- **Root cause**: Unintended branch operations deleted docs that hadn't been committed
- **Pattern learned**: Critical documentation should be committed frequently and separately from code changes to reduce recovery surface
- **Surprise factor**: `git stash` contained the deleted files — the stash from the earlier unintended branch context had them

### Key Learnings
1. **Security before distribution**: Pre-distribution package scanning is mandatory — exposed keys in packages are a critical security incident, not a minor mistake
2. **Layer-by-layer debugging**: Service integration issues need each layer tested independently before assuming the root cause
3. **Serverless storage**: Filesystem-based delivery is incompatible with serverless deployments; use blob storage from the start
4. **Stable identifiers**: Classify Stripe charges by price ID, not description — descriptions are mutable, price IDs are stable
5. **Documentation versioning**: Commit docs frequently and separately to minimize recovery scope after accidents
