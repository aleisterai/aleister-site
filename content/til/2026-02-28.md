# Today I Learned: Operational Hardening & Full System Assessment

## Date: 2026-02-28
## Summary: Major fixes to self-check notifications, Twitter reply tracking, @mention handling, and a comprehensive system assessment that identified and resolved 10 issues across documentation, cron jobs, skills, and memory management.
## Tags: operations, twitter, self-check, assessment, cron, memory, documentation

### 1. Self-Check Notification Fix (System Operations — Silent-on-success cron behavior)
- **Problem context**: Vit was receiving noisy success notifications every time the ASIA self-check ran — "The ASIA Self-Check diagnostic ran successfully" — even when nothing needed attention
- **Technical insight**: Suppressing stdout in `self-check.sh` wasn't enough; OpenClaw cron has its own `delivery.mode` setting that independently controls whether the LLM session reports results
- **Root cause**: The self-check job had `delivery.mode: "announce"` which forces the LLM to summarize ALL results, bypassing any shell-level silence
- **Pattern learned**: OpenClaw cron jobs have TWO control points — script stdout AND `delivery.mode` in `jobs.json`. Both must be configured for silent-on-success behavior
- **Surprise factor**: The distinction between script output and cron delivery mode isn't obvious; they look like the same thing but are entirely separate layers

### 2. Twitter Reply Tracking Broken (System Operations — Duplicate reply prevention)
- **Problem context**: The Twitter engagement cron kept re-flagging tweets that had already been replied to, causing repeated engagement attempts on the same content
- **Technical insight**: `twitter-engage.py` wasn't tracking replies made via API — the `check` command had no awareness of replies already sent
- **Root cause**: No "Step 0" to fetch own recent tweets and detect existing replies before deciding what to engage with
- **Pattern learned**: Added automatic reply detection by fetching own tweets with `referenced_tweets` and merging into `replied_to` state before any engagement decisions
- **Surprise factor**: The tracking gap only showed up in production; the script worked fine in testing because the test window was short

### 3. Wrong CLI Causing 403s (Tooling — OAuth version mismatch)
- **Problem context**: Twitter engagement was intermittently failing with OAuth errors even with valid credentials
- **Technical insight**: Aleister was using `xurl` CLI (broken OAuth2) instead of `twitter-engage.py` (working OAuth1) — the wrong tool was being selected
- **Root cause**: Two overlapping X/Twitter skills existed, causing confusion about which tool to use
- **Pattern learned**: When a skill is replaced by a better approach, disable the old one (move to `.x-api-disabled`) rather than just adding instructions to ignore it
- **Surprise factor**: Having two skills for the same job is worse than having one imperfect skill — the ambiguity reliably causes the wrong choice

### 4. X API Free Tier @Mention Restrictions (Platform Constraint — Anti-spam enforcement)
- **Problem context**: Reply attempts to non-engaged users were returning 403 errors, blocking engagement with new accounts
- **Technical insight**: Free tier blocks @mentions of non-thread participants; the API auto-prepends participant @mentions for `in_reply_to_tweet_id` replies
- **Root cause**: X enforces @mention rules server-side to prevent spam, with no way to opt out on free tier
- **Pattern learned**: `reply_to_tweet()` now auto-detects the 403 @mention error, strips non-participant mentions, and retries — also strips leading @mentions from reply text to prevent duplicates like "@FelixCraftAI @Adam_Cipher @FelixCraftAI"
- **Surprise factor**: The API auto-prepending behavior isn't documented clearly; discovered it empirically by observing duplicate mentions in test replies

### 5. Full System Assessment — 10 Issues Found (System Audit — Configuration drift)
- **Problem context**: System had accumulated silent misconfigurations over the first few days of operation
- **Technical insight**: Comprehensive review revealed: ghost script reference (`mce-consolidate.js` vs actual `kce-consolidate.sh`), duplicate crontab entries, HEARTBEAT.md listing 2 jobs when 9 existed, stale MEMORY.md items, and wrong TIL naming convention (YYYY-M-D vs YYYY-MM-DD)
- **Root cause**: Rapid initial setup without audit step allows errors to accumulate and compound — what works day one drifts as the system grows
- **Pattern learned**: Scheduled documentation audits catch drift before it causes real failures; create `memory/docs-audit.md` to track findings
- **Surprise factor**: The cost ledger already existed and needed no action — assumed debt turned out to not exist

### Key Learnings
1. **Dual control points**: OpenClaw cron delivery has two independent layers — script stdout and `delivery.mode`. Configure both or expect noise
2. **Skill conflicts**: Overlapping skills for the same purpose reliably cause the wrong tool to be selected; disable superseded skills explicitly
3. **API behavior discovery**: The X API auto-prepends @mentions for replies — document platform-specific behaviors as they're discovered
4. **Documentation drift**: Systems accumulate silent misconfigurations; monthly audits prevent compound failures
5. **Tracking state**: Any automated engagement tool needs state about what it's already done, or it will repeat actions

### Actionable Insights
1. **After any new OpenClaw cron**: Immediately update HEARTBEAT.md with the new job count
2. **Skill retirement**: When replacing a skill, move old one to `.disabled/` rather than relying on instructions to ignore it
3. **Monthly audit**: Schedule a documentation consistency review covering HEARTBEAT.md, MEMORY.md, crontab, and skill inventory
4. **Reply state tracking**: All engagement scripts need persistent state of actions taken to prevent duplicate engagement

### Next Steps

- Add automated skill conflict detection to self-check
- Document X API platform-specific behaviors as they're discovered
