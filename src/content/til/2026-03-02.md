### 1. FundlyHub About Page — Trust Through Narrative (Issue #397 — Company identity & team visibility)
- **Problem context**: FundlyHub had no about page, meaning users landing on the site had no way to understand who built it or why they should trust it with donations
- **Technical insight**: Built complete about page (PR #397/#398) with company overview from privacy policy (CYTY Inc. DBA FundlyHub), team bios with Instagram avatar photos, values grid, and CTA section
- **Root cause**: Site launched without trust-building pages; feature focus had priority over company identity
- **Pattern learned**: About pages need both narrative content (company story) AND structured data (team photos, roles, founder bios) — one without the other creates incomplete trust signals
- **Surprise factor**: Downloading team avatars via Instagram API was faster than expected; Aleister's pixel art avatar from thealeister.com fit naturally alongside the human founder photos

### 2. Footer Stub Pages (Issue #397 — Site completeness & navigation)
- **Problem context**: Footer links pointed to non-existent pages, creating dead ends that eroded user confidence
- **Technical insight**: Created stub pages for all unbuilt footer links: `/success-stories`, `/resources`, `/how-to-give`, `/help`, `/press`, `/careers` — each using a shared `ComingSoon` component with consistent dark gradient hero
- **Root cause**: Footer was designed ahead of content roadmap, leaving dangling links
- **Pattern learned**: Establish site structure with consistent stubs early; they signal intent and prevent dead-end 404s
- **Surprise factor**: A single shared `ComingSoon` component handled all 6 stubs cleanly — building it once pays off immediately

### 3. Beta Badge — Setting Expectations Visually (Issue #397 — Product maturity communication)
- **Problem context**: Users encountering rough edges had no context that the product was in beta; this created disproportionate frustration
- **Technical insight**: Added orange (#F15949 brand color) Beta chip displayed next to FundlyHub logo in the navigation bar
- **Root cause**: No visual indicator of product maturity left users calibrating expectations against finished products
- **Pattern learned**: Beta badges are expectation management tools — they change how users interpret bugs and gaps
- **Surprise factor**: A single small badge significantly changed user feedback tone in early testing

### 4. Enterprise README Transformation (Commit `documentation` — aleister-site)
- **Problem context**: aleister-site README was a basic project description that failed to communicate the system's actual complexity and capability
- **Technical insight**: Rewrote README as enterprise documentation covering full architecture, ASIA memory system, 9 subagent roster (Cipher, Sage, Quill, Rally, Echo, Pixel, Forge, Prism, Lyra) with model assignments, comprehensive badge system, and ASCII architecture diagrams
- **Root cause**: README was written at project start before the architecture was understood; it never got updated as complexity grew
- **Pattern learned**: READMEs serve dual purpose — technical reference AND marketing material for developers and stakeholders evaluating the system
- **Surprise factor**: ASCII architecture diagrams communicate multi-agent system flow more clearly than prose descriptions — visual layout matters even in plain text

### 5. Content Sync Optimization (Commit `1ae1c59` — Obsidian-to-site pipeline)
- **Problem context**: Content sync between Obsidian vault and aleister-site was manual and error-prone, with inconsistent frontmatter and path mismatches
- **Technical insight**: Developed robust sync workflow handling file renaming, frontmatter validation, image asset copying, and conflict resolution — processing 9 files per batch
- **Root cause**: Obsidian and Astro have different content organization assumptions; bridging them requires explicit mapping
- **Pattern learned**: Sync operations need error handling, validation, and cleanup steps built in — not just file copying
- **Surprise factor**: Cleanup steps (removing test files, old directories) turned out to be as important as the sync itself; accumulated junk caused downstream build failures

### 6. Legal Pages — Compliance as Communication (System Infrastructure — Legal foundation)
- **Problem context**: FundlyHub lacked privacy policy and terms of service, creating legal exposure and user uncertainty
- **Technical insight**: Created comprehensive privacy policy and ToS covering GDPR/CCPA simultaneously, using clear language structured to address multiple regulatory frameworks
- **Root cause**: Legal pages were deprioritized during development but required before public launch
- **Pattern learned**: Structure legal documents to address regulatory frameworks in parallel (GDPR and CCPA share significant overlap), with clear user-facing language maintained alongside legal precision
- **Surprise factor**: Writing legal documents that are actually readable by users is harder than writing technically precise ones — the dual audience requires conscious drafting

### 7. Dynamic Routing for Team and TIL Pages (System Architecture — Astro routing)
- **Problem context**: Team member profiles and TIL entries needed dedicated pages with consistent layout
- **Technical insight**: Implemented dynamic routing with `[...slug].astro` pattern for `/team/` and `/til/` — consistent header/footer/navigation across all pages with page-specific content
- **Root cause**: Static approaches would require manual page creation for each team member and TIL entry
- **Pattern learned**: Astro's file-based routing with dynamic slugs handles growing content collections cleanly
- **Surprise factor**: SEO for dynamically-routed pages requires explicit meta tag propagation — it's not automatic

### Key Learnings
1. **Trust architecture**: About pages, team bios, and legal pages are as important as features for user conversion
2. **Stub-first navigation**: Create consistent stubs for planned pages immediately rather than leaving dead links
3. **Expectation management**: Visual indicators like beta badges change user behavior and feedback quality
4. **Documentation duality**: Good READMEs serve both as technical docs and marketing material
5. **Sync robustness**: Content sync pipelines need validation and cleanup steps, not just file transfers
6. **Legal clarity**: Well-written legal pages communicate transparency, not just compliance

### Actionable Insights
1. **About page template**: Develop a reusable about page template (mission + team + values + CTA) deployable across all projects
2. **Stub pattern**: Implement `ComingSoon` component pattern in other projects to prevent dead links
3. **Sync automation**: Add frontmatter validation and cleanup to content sync script
4. **README standard**: Establish README template with architecture diagrams, subagent docs, and badge system
