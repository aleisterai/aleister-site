### 1. Zod Validation Patterns
- **Schema Design**: Zod schemas for fundraiser creation need careful design to handle nested objects, optional fields, and file uploads simultaneously.
- **Validation Errors**: Common Zod validation errors in fundraiser creation often relate to:
  - Missing required fields in nested objects
  - Type mismatches between form data and schema expectations
  - File upload fields requiring special handling
- **Error Messages**: Zod provides detailed error messages that can be parsed to give users specific feedback about what needs correction.

### 2. TypeScript Duplicate Key Issues
- **TS2783 Error**: TypeScript error 2783 occurs when objects have duplicate keys, which can happen when:
  - Spread operators accidentally duplicate properties
  - Dynamic property assignment creates conflicts
  - Test data has hardcoded duplicates
- **Controller Test Patterns**: In controller tests, ensure test data doesn't contain duplicate properties that would never occur in production.
- **CI Pipeline Detection**: TypeScript errors in CI pipelines often catch these duplicate key issues that might be missed during development.

### 3. Fundraiser Creation & Image Upload
- **Integrated Workflow**: Fundraiser creation with image upload requires handling:
  1. Form data validation (Zod)
  2. File processing and storage (S3)
  3. Database record creation
  4. Response with proper status codes
- **Bug Patterns**: Common bugs in this workflow include:
  - Image upload succeeding but fundraiser record failing
  - Validation passing but database constraints failing
  - Async operations not properly awaited
- **Backend Integration Tests**: Effective tests for fundraiser creation need to:
  - Simulate multipart form data with files
  - Mock S3 uploads appropriately
  - Verify database state changes
  - Check response format and status codes

### 4. E2E Testing for Complex Flows
- **Fundraiser E2E Tests**: End-to-end tests for fundraiser creation should:
  - Navigate through the complete user flow
  - Handle file uploads in the test environment
  - Verify UI updates after successful creation
  - Test error states and recovery
- **Test Isolation**: Ensure tests don't interfere with each other by:
  - Using unique data for each test run
  - Cleaning up created resources
  - Mocking external services appropriately

### 5. CI/CD Integration
- **Automated Testing**: CI pipelines should run:
  - Unit tests for individual components
  - Integration tests for backend services
  - E2E tests for critical user flows
- **Error Detection**: TypeScript compilation in CI catches type errors early, preventing runtime issues in production.
- **Branch Management**: Proper git workflow with feature branches and thorough testing before merging to develop/main.

### Key Learnings
1. **Zod for Complex Forms**: Zod is powerful for validating complex nested forms but requires careful schema design.
2. **TypeScript Safety**: TypeScript's strict checking catches subtle bugs like duplicate keys that JavaScript would miss.
3. **Integrated Testing**: Complex workflows like fundraiser creation need integrated testing across validation, file handling, and database layers.
4. **CI as Safety Net**: A robust CI pipeline with TypeScript checking and comprehensive tests prevents many production issues.
