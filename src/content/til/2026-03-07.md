### 1. Fundraiser Creation Failing Silently — Zod Schema Bug (Issue #440 — Image upload & fundraiser creation)
- **Problem context**: New fundraiser creation in production was failing with a generic "Error creating fundraiser" message; users couldn't create new fundraisers at all
- **Technical insight**: The Zod schema had `owner_user_id: z.string().uuid()` but the controller was passing a numeric `profileId` (integer from profiles table), causing `NaN` coercion that failed validation silently
- **Root cause**: Schema was written expecting UUID strings but the data pipeline was passing integer IDs — a type contract violation invisible at compile time without Zod
- **Pattern learned**: Commit `c9630791` fixed the core bugs; `7f0f5db2` added backend integration + E2E tests to catch regressions. Zod schemas must match actual data types flowing through controllers, not assumed types
- **Surprise factor**: The Zod validation error was masked by a separate `403 email_verified` bug; two independent failures compounded, making diagnosis significantly harder

### 2. Fundraiser Image Upload Broken (Issue #440 — Production data loss)
- **Problem context**: Users could submit the fundraiser creation form with an image but the image wasn't being saved or displayed
- **Technical insight**: Image upload and fundraiser record creation were happening in the wrong order; image upload success didn't guarantee fundraiser record creation
- **Root cause**: Async operations weren't properly coordinated — the image upload completed but if the fundraiser creation failed, the image was orphaned with no error propagation
- **Pattern learned**: Multi-step create flows need atomic error handling: if any step fails, all prior steps must be rolled back or the user must be clearly informed of partial state
- **Surprise factor**: The image upload was succeeding at the S3 level — so the bug appeared intermittent from the frontend perspective while being systematic from the data perspective

### 3. TypeScript Duplicate Key Detection — TS2783 (Issue #442 — CI pipeline quality)
- **Problem context**: CI pipeline was failing on TypeScript compilation with TS2783 errors in controller tests
- **Technical insight**: Commit `2b2af6b6` removed duplicate `slug` property in controller test; commit `52a03723` fixed the TS2783 duplicate key error in CI. TypeScript error TS2783 triggers when object literals have duplicate keys
- **Root cause**: Test data was manually constructed and had a `slug` property defined twice — JavaScript silently uses the last value, TypeScript correctly rejects it
- **Pattern learned**: TypeScript's strict checking catches duplicate key bugs that JavaScript allows silently; CI TypeScript compilation is a valuable safety net
- **Surprise factor**: The duplicate key worked at runtime (JavaScript took last value), so the feature appeared functional in manual testing — only CI caught it

### 4. Backend Integration Tests for Fundraiser Creation (Commit `7f0f5db2` — Test coverage)
- **Problem context**: No backend integration tests existed for the fundraiser creation flow; bugs were only caught in production
- **Technical insight**: Added 25 new backend integration tests (`fundraiser-creation-flow.test.ts`) covering the full creation flow with multipart form data, S3 mock, database state verification, and response format validation
- **Root cause**: Unit tests for individual components existed but full flow integration wasn't tested
- **Pattern learned**: Effective integration tests for create flows must: simulate multipart form data, mock S3 appropriately, verify database state changes, and check response format AND status codes
- **Surprise factor**: Writing the tests revealed 3 additional edge cases that weren't in the original bug report — empty optional fields, duplicate submissions, and malformed image data

### 5. E2E Tests for Fundraiser Creation Wizard (Commit `7f0f5db2` — End-to-end coverage)
- **Problem context**: The fundraiser creation wizard had a multi-step UI flow that integration tests couldn't fully validate
- **Technical insight**: Added 10 Playwright E2E tests for the creation wizard covering full navigation, file uploads in test environment, UI state verification, and error state recovery
- **Root cause**: Wizard UX bugs only appear when the full DOM interaction sequence is executed, not in API-level tests
- **Pattern learned**: E2E tests for wizard flows must use unique data per test run, clean up created resources, and mock external services — test isolation prevents cascading failures
- **Surprise factor**: File upload in Playwright requires explicit `setInputFiles()` — standard form interactions don't trigger file input events correctly

### 6. Fundraiser Zod Schema Fix — TS2783 in Production (Issue #442 — Backend validation)
- **Problem context**: After fixing issue #440, a new fundraiser creation Zod validation error appeared specifically around `owner_user_id`
- **Technical insight**: PR #442 description stated `owner_user_id` needed to be `z.number().int().positive()` (numeric profile ID), but earlier the field had been changed to `z.string().uuid()` — conflicting fixes for different parts of the same problem
- **Root cause**: The identity type (UUID vs integer) was inconsistent across different parts of the codebase; fixing it in one place revealed the mismatch in another
- **Pattern learned**: Schema type changes require simultaneous updates to ALL consuming code: Zod schema, TypeScript types, controllers, services, and test fixtures — partial updates create new bugs
- **Surprise factor**: The fix in issue #440 and the fix in issue #442 applied changes to the same field in opposite directions — revealing a deeper inconsistency in how profile IDs were used throughout the system

### Key Learnings
1. **Compound failures hide root cause**: When two independent bugs coexist, diagnosing either one is harder — fix the noisiest failure first to expose the underlying one
2. **Async create flows need atomicity**: Multi-step creation (form → upload → save) must be coordinated; partial success is worse than clean failure
3. **TypeScript catches what JavaScript misses**: Duplicate keys, type mismatches, and null safety issues that JavaScript allows silently are caught by TypeScript in CI
4. **Integration tests reveal edge cases**: Writing tests for complex flows uncovers scenarios not in the original bug report
5. **Schema type changes cascade**: Changing a field's type requires simultaneous updates everywhere it's used

### Actionable Insights
1. **Atomic create pattern**: All multi-step creation flows (form + upload + database) must implement rollback on partial failure
2. **Test data factory**: Implement a test data factory that generates valid, unique, type-safe test data without duplicate keys
3. **CI TypeScript strict mode**: Ensure all projects run TypeScript in strict mode during CI to catch type mismatches early
4. **Schema change protocol**: When changing a Zod schema field type, grep for all usages and update them atomically in the same PR
