### 1. Authentication Identity Resolution (Commit: `322ea572`)
- **Critical Bug: userId vs cognitoSub Mismatch**: Discovered in fundlyhub commit `322ea572` where `updateFundraiser` and `deleteFundraiser` used `req.user?.userId` (profile UUID) when `checkOwnership()` expected the raw Cognito JWT `sub` claim, causing `isOwner` to always return `false` → instant 403 Forbidden.
- **Root Cause Analysis**: The auth middleware returns both `userId` (database UUID) and `cognitoSub` (Cognito subject ID), but ownership checks must use `cognitoSub` for comparison with stored ownership records.
- **Identity Resolution Pattern**: Learned the proper pattern:
  1. Extract `cognitoSub` from auth token (JWT `sub` claim)
  2. Look up user profile using `cognitoSub`
  3. Use profile's `id` (UUID) for database operations
  4. Never mix authentication IDs with database IDs
- **Fix Implementation**: Switched both handlers to use `req.user?.cognitoSub` for ownership checks while keeping `req.user?.userId` in log messages for readability.

### 2. Progressive Web App (PWA) Implementation (Commit: `2deac16f`)
- **Apple Touch Icon Generation**: Learned from commit `2deac16f` that Safari ignores SVG for `apple-touch-icon` and requires optimized PNG icons at specific sizes (120x120, 152x152, 180x180, 192x192, 512x512 pixels).
- **Icon Optimization Workflow**: Used Pillow LANCZOS resampling + optimization to generate high-quality PNG icons from source `color-logo-icon.png`.
- **Web App Manifest Creation**: Created `manifest.webmanifest` with:
  - App name and short name (`FundlyHub`)
  - Theme color (`#6C5CE7`) matching brand
  - Display mode (`standalone`) for app-like experience
  - Comprehensive icon definitions for all device sizes
- **PWA Meta Tags**: Added `apple-mobile-web-app` meta tags and `theme-color` meta tag for native iOS feel.
- **Key Insight**: PWA implementation provides significant UX improvement (home screen installation, offline capabilities) with relatively minimal effort once the icon generation workflow is established.

### 3. UUID Schema Alignment (Commit: `7e5f5b47`)
- **Critical Schema Mismatch**: Discovered in commit `7e5f5b47` that Zod schema had `owner_user_id: z.number().int().positive()` while the database column was UUID type, causing auth middleware UUID strings to be coerced to `NaN` by `Number()`.
- **Root Cause Analysis**: The `400 Bad Request` error was masked by the `403 email_verified` bug, making debugging difficult.
- **Fix Implementation**:
  1. Updated Zod schema: `owner_user_id → z.string().uuid()`
  2. Added `goal_amount → z.coerce.number()` for form input safety
  3. Updated `fundraiser.service.ts`: `profileId: number → string`
  4. Removed `Number()` coercion in controllers, using UUID directly
  5. Updated all test fixtures to use UUID strings
- **Database Schema Consistency Rule**: When migrating from integer IDs to UUIDs, ALL related schemas, services, and test fixtures must be updated simultaneously to prevent type mismatches.

### 4. SSO Email Verification Handling (Commit: `fa788253`)
- **Cognito OAuth Limitation**: Learned from commit `fa788253` that Cognito does NOT set `email_verified=true` for federated (OAuth) users, even with valid Google/Apple ID tokens.
- **Enterprise Policy Decision**: Established that Google/Apple OAuth + valid ID token = email verified, regardless of Cognito's `email_verified` claim.
- **Fix Implementation**:
  1. `oauth.controller.ts`: Force `emailVerified=true` for OAuth provisioning
  2. `core.controller.ts` `/me` endpoint: Detect OAuth from `cognito:username` pattern (`google_xxx`, `signinwithapple_xxx`) and set `effectiveEmailVerified=true`
  3. Auto-repairs existing profiles on next page load
- **Verification Confirmation**: Verified via Cognito console that `testerfundly@gmail.com` has `email_verified=No` as expected for federated users, confirming the need for this workaround.

### 6. Fundraiser Schema Design
- **Empty String Handling**: Zod schemas for fundraiser creation should accept empty strings in optional fields and convert them to `null`/`undefined`.
- **Optional Field Patterns**: Differentiate between:
  - Required fields (validation fails if missing)
  - Optional fields (can be `null`, `undefined`, or empty string)
  - Default-valued fields (use default if not provided)
- **Schema Validation Order**: Apply transformations (empty string → `null`) before validation to avoid schema errors.

### Key Learnings
1. **Identity Resolution Critical**: Mixing authentication IDs (`cognitoSub`) with database IDs (`userId`) causes subtle, hard-to-debug 403 errors that break core functionality.
2. **PWA Simple Wins**: Basic PWA features (proper PNG icons, web manifest) provide significant UX improvements with minimal implementation effort.
3. **UUID Consistency Mandatory**: Once you choose UUIDs, enforce consistency across Zod schemas, TypeScript types, database columns, and test fixtures simultaneously.
4. **SSO Trust Model**: Trust major identity providers (Google/Apple) email verification regardless of Cognito's `email_verified` claim for federated users.
5. **Repository Commit Analysis**: Critical learnings live in git commits; must systematically check all active repositories before creating TIL documents.
